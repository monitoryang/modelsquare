# 需求规格说明书 - 置信度与 IOU 实时调整功能

**版本**: v1.3  
**日期**: 2025-07-11  
**状态**: 待确认

---

## 1. 项目概述

### 1.1 背景

ModelSquare 是一个 AI 模型推理平台，支持两种视频推理模式：

- **视频测试**：用户上传视频文件，后端完成目标检测推理后在结果页通过 `VideoPlayer` 组件在前端 canvas 叠加检测框展示
- **实时推流**：用户通过 RTMP 向平台推送实时视频流，后端实时进行目标检测，前端通过 WebSocket 接收检测结果并叠加显示

**核心设计理念（新方案）：**

采用**后端零过滤 + 前端动态过滤**的架构：

- **后端**：推理时将置信度阈值设为 0、IOU 阈值设为 0（或极低值），将**所有**检测框及其原始分数（`scores`）传给前端，不在服务端过滤
- **前端**：根据用户滑块设置的 confidence 和 IOU 阈值，对接收到的检测框数据进行**本地实时过滤**，只渲染满足条件的检测框

此方案的优势：参数调整无需重新推理，即刻生效，响应极快。

**当前状态：**

| 场景 | 检测框渲染 | 参数调整 |
|------|-----------|---------|
| 实时推流（`StreamTest.tsx`）| 现有 canvas 叠加层逻辑存在，但参数联动过滤未实现 | 推流激活后滑块被禁用，无法调整 |
| 视频测试结果（`ModelDetail/index.tsx` + `VideoPlayer`）| `VideoPlayer` 已实现 canvas 叠加，但过滤逻辑仅支持类别过滤，未支持置信度/IOU 过滤 | 无调整控件 |
| 个人中心预览（`Profile/index.tsx` + `VideoPlayer`）| 同上 | 同上 |

### 1.2 目标

1. **推理侧改造**：后端推理（视频推理 + 实时推流）统一以低阈值（conf=0，iou=0）执行，返回完整的原始检测结果（含所有候选框及分数）
2. **前端过滤实现**：前端 `VideoPlayer` 和推流 canvas 叠加层增加基于 confidence/IOU 的本地过滤逻辑
3. **参数控件添加**：三个场景均增加置信度和 IOU 调整滑块，用户拖动后立即更新检测框显示

### 1.3 范围

**包含：**

- 修改后端视频推理和推流推理，以低阈值（conf=0，iou=0）输出完整原始检测结果
- 前端 `VideoPlayer` 组件增加 confidence/IOU 过滤逻辑
- 前端推流 canvas 叠加层增加 confidence/IOU 过滤逻辑
- 三个场景均增加置信度和 IOU 调整滑块
- 参数联动的检测框数量实时更新

**不包含：**

- 图片推理（不在本次范围内）
- 推理参数的持久化保存（用户偏好设置）
- API 调用场景下的参数调整

## 2. 功能需求

### 2.1 用户角色

| 角色名称 | 描述 | 权限 |
|----------|------|------|
| 已登录用户 | 使用平台进行模型推理测试的用户 | 调整自己会话/任务的参数、查看检测结果 |

### 2.2 功能清单

#### 2.2.1 后端推理低阈值输出（视频推理）

- **需求ID**: FR-001
- **需求描述**: 视频推理后端（`video_inference.py`）在执行推理时，将 conf_threshold 和 iou_threshold 设置为 0（或极小值如 0.001），使推理结果包含所有候选检测框及其原始分数，不在服务端过滤。前端根据用户设置的阈值进行本地过滤。
- **优先级**: 高
- **验收标准**:
  - 视频推理时以 conf=0、iou=0 调用推理引擎
  - 每帧的 `frame_results` 中包含所有候选框及其原始 `scores`（不被服务端阈值截断）
  - `VideoTaskResult` 结构不变，仅数据量增大（包含更多低分框）
- **依赖关系**: 无

#### 2.2.2 后端推理低阈值输出（实时推流）

- **需求ID**: FR-002
- **需求描述**: 实时推流后端（`stream_inference.py`）在执行推理时，同样使用低阈值（conf=0，iou=0），将完整原始检测框通过 WebSocket 推送给前端。前端负责过滤。
- **优先级**: 高
- **验收标准**:
  - 推流推理以 conf=0、iou=0 调用推理引擎
  - WebSocket 推送的 `inference_result` 包含所有候选框及原始分数
  - 现有 `StreamSession.conf_threshold / iou_threshold` 字段可保留（用于记录用户偏好），但不再影响推理引擎调用参数
- **依赖关系**: 无

#### 2.2.3 VideoPlayer 前端置信度/IOU 过滤

- **需求ID**: FR-003
- **需求描述**: `VideoPlayer` 组件（`components/VideoPlayer/index.tsx`）增加 confidence 和 IOU 阈值的本地过滤逻辑。当用户拖动滑块时，组件实时重新过滤当前帧的检测框并重绘 canvas，无需任何网络请求。
- **优先级**: 高
- **验收标准**:
  - `VideoPlayer` 接收 `confThreshold` 和 `iouThreshold` 两个新 props（或提供内部状态管理）
  - `getFilteredDetections` 函数在现有类别过滤基础上，增加对 `score >= confThreshold` 的过滤
  - IOU 过滤：对同一帧内同类别的检测框进行 NMS（Non-Maximum Suppression）过滤，去除与高分框 IOU 超过阈值的低分重叠框
  - 滑块变化时，canvas 叠加层实时重绘，无需重新加载视频
  - 当前帧满足条件的检测框数量实时显示在 canvas 左上角（已有 `检测: N 个` 显示）
- **依赖关系**: FR-001（需要后端提供低阈值原始数据）

#### 2.2.4 视频测试结果页增加参数调整控件

- **需求ID**: FR-004
- **需求描述**: 在 `ModelDetail/index.tsx` 视频测试标签页的结果展示区域（`VideoPlayer` 上方或内部），增加置信度和 IOU 调整滑块，滑块变化实时触发 `VideoPlayer` 重新过滤。
- **优先级**: 高
- **验收标准**:
  - 视频任务完成后，`VideoPlayer` 上方/下方显示置信度和 IOU 滑块，默认值为常用推荐值（普通模型 conf=0.25、iou=0.45；OWL 模型 conf=0.1、iou=0.3）
  - 拖动滑块时，`VideoPlayer` 检测框立即响应（实时过滤，无网络请求）
  - 滑块旁实时显示当前帧满足条件的检测框数量
- **依赖关系**: FR-001、FR-003

#### 2.2.5 实时推流前端置信度/IOU 过滤

- **需求ID**: FR-005
- **需求描述**: `StreamTest.tsx` 推流 canvas 叠加层的 `drawDetections` 函数，在绘制检测框时增加基于 confidence 和 IOU 阈值的本地过滤逻辑；推流激活后滑块解除禁用，用户拖动即时生效。
- **优先级**: 高
- **验收标准**:
  - 推流激活后，置信度和 IOU 滑块处于可操作状态（移除 `disabled={streamSession?.status === 'active'}`）
  - `drawDetections` 调用时，根据当前 `confThreshold` / `iouThreshold` 过滤传入的检测框，只绘制满足条件的框
  - 滑块变化即刻生效（前端本地过滤，无需请求后端），下一帧绘制时使用新阈值
  - 实时显示当前渲染的检测框数量
- **依赖关系**: FR-002（需要后端提供低阈值原始数据）

#### 2.2.6 个人中心预览增加参数调整控件

- **需求ID**: FR-006
- **需求描述**: `Profile/index.tsx` 的视频预览弹窗中，在 `VideoPlayer` 旁增加置信度和 IOU 调整滑块，与 FR-004 功能一致。
- **优先级**: 中
- **验收标准**:
  - 预览弹窗中包含置信度和 IOU 滑块
  - 拖动滑块时，`VideoPlayer` 检测框实时响应（本地过滤）
  - 滑块旁实时显示当前帧满足条件的检测框数量
- **依赖关系**: FR-001、FR-003

**以便于** 实时过滤显示的检测框，无需停流即可看到不同灵敏度下的效果

**验收条件**:

- 推流处于 `active` 状态时，参数滑块可以被操作
- 滑块调整后，下一帧绘制时立即使用新阈值过滤（无需等待后端响应）

## 4. 数据需求

### 4.1 架构变化

**后端数据输出变化：**

| 场景 | 变更前 | 变更后 |
|------|--------|--------|
| 视频推理 | 按用户传入 conf/iou 过滤后存储 `frame_results` | 以 conf=0、iou=0 推理，存储全量原始候选框及 scores |
| 实时推流 | 按 session.conf_threshold/iou_threshold 过滤后推送 | 以 conf=0、iou=0 推理，推送全量原始候选框及 scores |

**前端数据处理变化：**

| 组件 | 变更前 | 变更后 |
|------|--------|--------|
| `VideoPlayer` | 仅支持类别过滤 | 增加 conf/IOU 本地过滤（NMS） |
| `StreamTest.tsx` drawDetections | 直接绘制后端已过滤的框 | 按前端 conf/iou 参数过滤后再绘制 |

### 4.2 数据流

**视频播放参数调整**（前端本地，无网络请求）：

```
用户拖动滑块 → confThreshold / iouThreshold 状态更新
→ VideoPlayer.getFilteredDetections()：
    对当前帧 frame_results 中的框按 score >= confThreshold 过滤
    对通过分数过滤的框执行前端 NMS（去除 IOU 超阈值的低分重叠框）
→ drawOverlayOnCanvas() 重绘 canvas → 检测框数量提示更新
```

**实时推流参数调整**（前端本地，无网络请求）：

```
用户拖动滑块 → confThreshold / iouThreshold 状态更新
→ WebSocket 新帧数据到达 → drawDetections()：
    对接收到的全量候选框按 score >= confThreshold 过滤
    执行前端 NMS → 绘制满足条件的框
    检测框数量提示更新
```

## 5. 假设和依赖

### 5.1 假设

- **低阈值推理性能**：以 conf=0 推理会返回更多候选框，需评估对推理耗时和数据传输量的影响；若性能不可接受，可设置一个极低阈值（如 conf=0.01）作为下限，而非严格的 0
- **前端 NMS 实现**：前端需实现简单的 NMS 算法（按分数排序 + IOU 计算），用于 IOU 阈值过滤；`VideoPlayer` 和 `StreamTest.tsx` 中均需实现或共享此函数
- **参数默认值**：初始显示时建议使用常用推荐值（普通模型 conf=0.25、iou=0.45；OWL 模型 conf=0.1、iou=0.3），以便用户看到常规效果
- **参数边界**：confidence 和 IOU 均在 [0, 1] 之间，步长 0.05

### 5.2 依赖

- `VideoPlayer` 组件已有 `getFilteredDetections` 函数，需在此基础上扩展 conf/IOU 过滤逻辑
- 前端 NMS 算法可作为工具函数放在 `frontend/src/utils/` 中，供 `VideoPlayer` 和 `StreamTest.tsx` 共用
- 后端视频推理结果数据结构（`FrameDetectionResult`）无需改变，`scores` 字段已存在，仅数据量会增加
- 后端推流 WebSocket 推送的 `inference_result.detections.scores` 字段已存在
